Why the Xicar user app is signed, where the signing key lives on disk, how the key fingerprints are used by Firebase, and how to back up, hand off, or rotate the key when someone joins, leaves, or suspects compromise.

This document covers the **Xicar user app** (`com.xicartaxi.xicar_user`). The driver app and admin app follow the same pattern with their own keystores.

---

## Background

Every Android app distributed outside the device must be **digitally signed**. The signing key:

- Authenticates the app's author (you) — Play Store refuses unsigned APKs/AABs
- Forms the **update chain** — users can only install updates signed with the same key as the version already on their device
- Identifies the app to Google's APIs (Firebase, Maps, Play Integrity) via its **SHA-1** and **SHA-256** fingerprints

### Play App Signing model (since 2021)

| Key | Who holds it | What it signs |
| --- | --- | --- |
| **App signing key** | Google Play | The final APK/AAB that ships to users |
| **Upload key** (our `upload-keystore.jks`) | You, in this repo | The AAB you upload to Play Console |

<Note>
  **Xicar's Play App Signing is already enrolled.** The CI workflow signs the AAB with the upload key on every run; Play re-signs with the app signing key before distribution.
</Note>

## The keystore file

| Property | Value |
| --- | --- |
| Path | `android/app/upload-keystore.jks` |
| Alias | `xicar` |
| Algorithm | RSA 2048, signature `SHA256withRSA` |
| Validity | 25 years from generation date |
| Format | PKCS12 (Java 17 `keytool` default) |
| In git | **No** — covered by `*.jks` in `.gitignore` |

## `android/key.properties`

Gradle reads this file to sign release builds:

```properties
storeFile=app/upload-keystore.jks
storePassword=<store-password>
keyPassword=<key-password>
keyAlias=xicar
```

`key.properties` is **gitignored** and recreated on every CI run from the `ANDROID_KEY_PROPERTIES` GitHub secret. See [Secrets](/ci-cd/secrets).

## Getting the SHA-1 and SHA-256

### Release keystore (production builds)

```bash
keytool -list -v \
  -keystore android/app/upload-keystore.jks \
  -alias "$(grep '^keyAlias=' android/key.properties | cut -d= -f2)" \
  -storepass "$(grep '^storePassword=' android/key.properties | cut -d= -f2)" \
  | grep -E "SHA1:|SHA256:"
```

### Debug keystore (development builds)

```bash
keytool -list -v \
  -keystore ~/.android/debug.keystore \
  -alias androiddebugkey \
  -storepass android \
  | grep -E "SHA1:|SHA256:"
```

Register fingerprints in Firebase Console → **Project Settings** → **Your apps** → Android app row → **Add fingerprint**.

## Backup

**The keystore file and its password must be backed up outside the repo.** Recommended storage:

- **1Password / Bitwarden** — team vault with base64 of the `.jks` and the password
- **Offline backup** — encrypted USB drive in a safe

---

## Handoff / onboarding a new dev

1. Share the base64 of `android/app/upload-keystore.jks` and the password over a secure channel
2. Save keystore to `android/app/upload-keystore.jks`
3. Save `key.properties` to `android/key.properties`
4. Run a clean release build to confirm the setup
5. Compare the resulting AAB's signature against the expected SHA-256

## Rotation

Rotate the upload key when the key or password has leaked, a team member with access leaves, or the keystore file is lost.

### How to rotate

1. **Generate a new keystore** with `keytool -genkey`
2. **Update `android/key.properties`** with the new password
3. **Re-register the new fingerprints** in Firebase Console
4. **Notify Google Play** if replacing the upload key (Play Console → App integrity → Request upload key reset)
5. **Re-rotate GitHub secrets** `ANDROID_KEYSTORE_BASE64` and `ANDROID_KEY_PROPERTIES` — see [Secrets](/ci-cd/secrets)

### Release build works locally but fails on CI

The CI runs use the `ANDROID_KEYSTORE_BASE64` secret to materialize `android/app/upload-keystore.jks` on the runner. If the secret is stale, re-encode the local keystore and update the secret:

```bash
base64 -i android/app/upload-keystore.jks | tr -d '\n' | pbcopy
```

Also verify `ANDROID_KEY_PROPERTIES` matches your local `key.properties`.
