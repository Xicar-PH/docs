How CI distributes **dev** and **beta** builds to testers via Firebase App Distribution on a **separate** Firebase project from runtime.

## Two Firebase projects

|  | **xicarv2** | **xicarph---builds** |
| --- | --- | --- |
| Purpose | Runtime (Auth, Firestore, etc.) | App Distribution only |
| Project number | (existing) | `539047008498` |
| CI materializes | `google-services.json`, `GoogleService-Info.plist`, `.env` FB keys | Nothing at build time |
| CI uploads to | ‚Äî | Signed APKs (dev/beta) and IPAs (dev/beta) |

**Build with `xicarv2` config, distribute via `xicarph---builds`.**

Prod builds are **not** uploaded to Firebase App Distribution ‚Äî CI produces APK/IPA artifacts only.

## Tester groups

Create in Firebase Console ‚Üí **App Distribution** ‚Üí **Testers & Groups**:

| Group | Used by |
| --- | --- |
| `general-xicar-internal` | All distributable flavors (`dev` and `beta`) |

Add tester emails to each group. Testers install via the Firebase App Distribution app or email invite link.

---

## Service account (CI uploads)

1. Google Cloud Console ‚Üí project **xicarph---builds**
2. **IAM & Admin** ‚Üí **Service accounts** ‚Üí create `ci-app-distribution`
3. Role: **Firebase App Distribution Admin**
4. Generate JSON key ‚Üí GitHub secret `FIREBASE_APP_DIST_SERVICE_ACCOUNT_JSON`
5. Enable **Firebase App Distribution API** if not already enabled

### Android registration notes

When adding Android apps in `xicarph---builds`, **skip SHA-1/SHA-256 fingerprints**. Those belong on `xicarv2` for runtime Auth. The distribution project only receives APK uploads.

## How CI uses it

Fastlane lane `firebase_app_distribution` (via `fastlane-plugin-firebase_app_distribution`):

- **dev / beta** ‚Üí upload after build
- **prod** ‚Üí skip distribution; artifacts only

Local equivalent:

```bash
export FLAVOR=beta
# materialize secrets locally first
bundle exec fastlane android ci
# or
bundle exec fastlane ios ci
```
